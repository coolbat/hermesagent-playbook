The first workflow is where Hermes stops being "installed software" and starts becoming a useful system.

## Pick a workflow with low ambiguity

Good starter workflows share the same properties:

- short input
- short runtime
- obvious output
- easy manual review

Examples:

- summarize a link
- turn notes into a structured recap
- generate a daily briefing from a small source set
- route a short status update into one channel

## Why this matters

Your goal is not to prove Hermes can do everything. Your goal is to prove one entire loop works:

1. input arrives
2. Hermes uses the right capabilities
3. output lands in the right place
4. you can inspect the result

That tells you much more than a long, flashy demo.

## Start with built-in capability before optional complexity

Before you install extra skills from the Hub, ask whether built-in Hermes behavior is already enough for the first test. Many first workflows can be validated with:

- the CLI
- one configured provider
- a narrow prompt
- a known output path

Optional skills make more sense once you know exactly what gap you are filling.

## What to review after the first run

After the workflow returns a result, inspect:

- Was the result actually useful?
- Did Hermes use the right context?
- Could you explain why it succeeded?
- Would you know where to look if it failed next time?

If you cannot answer the last two questions, the workflow is still too big.

## A better first success metric

Do not ask "Was that impressive?"

Ask:

- Can I run it again?
- Can I debug it?
- Can I tell where the output should go?

That is the beginning of a reliable Hermes workflow practice.
