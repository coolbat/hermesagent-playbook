Most early Hermes failures are not mysterious. They usually belong to one of four layers:

- installation
- provider configuration
- gateway or platform routing
- workflow scope

## The shortest recovery loop

When something breaks, run the shortest useful sequence first:

```bash
hermes setup
hermes model
hermes doctor
```

If Hermes works in the CLI but not in a platform, add:

```bash
hermes gateway
```

This separates "the runtime is broken" from "the platform path is broken."

## Symptom-first debugging

Start from the symptom, not from your favorite guess.

Common examples:

- no CLI response: check install and provider
- CLI works, platform silent: check gateway and platform routing
- replies are weak or unstable: check model selection and workflow scope
- the wrong action happens: inspect skills, rules, and prompt boundaries

## What not to do

Avoid these early-debug mistakes:

- switching providers and channels at the same time
- installing new skills during diagnosis
- changing prompts before proving the base loop
- turning one failure into a full rebuild

Each extra variable makes the signal worse.

## Use Hermes diagnostics before theory

If Hermes gives you a diagnostic tool, use it before you speculate. Official guidance is intentionally operational:

- confirm environment state
- confirm provider state
- confirm routing state
- only then redesign the workflow

That order saves time.

## When to go deeper

Leave Quick Start once you can describe the failure precisely:

- "provider auth fails"
- "gateway runs but Telegram is silent"
- "workflow returns low-value output"

At that point the problem is specific enough for the Troubleshoot and Learn libraries to help.
