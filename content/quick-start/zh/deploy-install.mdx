第一次安装的目标，不是把 OpenClaw 打造成完美生产环境，而是先确认它**真的能启动、能调用模型、能留下排错证据**。只要这三件事成立，安装阶段就算合格。

## 什么叫“够用了”

对新手来说，一套“够用”的安装，至少要能回答下面三个问题：

- 进程能不能正常启动？
- 一个模型服务能不能认证成功？
- 出错时能不能看到日志？

如果这三件事都能确认，就已经足够进入下一步。后面的多渠道、多技能、远程网关、自动任务，都应该建立在这个基础上。

## 安装前的最低准备

不要一上来就翻十几篇教程。先把最小前提补齐：

- 一台你能持续使用的机器
- 正常的命令行环境
- 满足要求的 Node.js 版本
- 一个可用的模型提供商凭证
- 一点点耐心，因为第一次安装经常卡在环境细节上

如果你是第一次接触这类工具，建议把第一套环境装在你最熟悉的设备上。不要一边学 OpenClaw，一边学陌生操作系统、一边折腾远程服务器，这会把问题层层叠加。

## 安装顺序建议

1. 先选最简单的路径，通常是本地。
2. 先装本体，再确认命令能跑。
3. 只接一个模型服务，并立刻验证凭证。
4. 跑一次最短健康检查，确认日志可见。
5. 在基础健康检查通过前，不要继续叠加额外服务。

> 每多加一个还没验证过的依赖，排错成本都会上升。

## 推荐的最小验证动作

安装完后，不要马上去配飞书、装十几个技能，先做这类短动作：

- 查看 CLI 是否可用，比如帮助信息能否正常输出
- 跑一次初始化或 onboarding 流程
- 验证模型认证是否成功
- 看一眼配置文件和工作区是否真的生成
- 看一眼状态或日志，确认不是“静默失败”

如果你已经能完成这些动作，就说明安装链路基本健康。这个阶段最重要的是建立可观测性，而不是堆功能。

## 安装阶段最该避免的 4 件事

下面这些事，都会让第一次安装难度显著上升：

- 一开始就上远程部署或复杂网络拓扑
- 同时接多个 provider，出错后分不清是谁的问题
- 安装还没验证就批量装技能
- 看到报错后连续改环境、配置、模型、渠道

第一次安装最怕的不是慢，而是变量太多。

## 遇到安装报错时，先判断是哪一层

大多数安装问题都属于下面四层之一：

- 环境层：Node 版本、系统依赖、路径问题
- 网络层：下载失败、代理问题、仓库拉取失败
- 凭证层：模型 key 无效、provider 配错
- 配置层：初始化没完成、配置文件不一致

先分层，再排错，比直接搜索报错字符串更高效。因为你真正需要的不是“碰巧有人遇到过同样的字面错误”，而是知道问题落在哪一段链路里。

## 新手最常见的误区

很多人会把 setup 当成架构设计题，在第一个有用工作流出现之前，就花大量时间优化环境、切换部署方式、讨论最佳实践。结果是安装投入很多，系统价值却还没被证明。

## 更好的下一步

当基础环境能稳定启动后，就进入**接通一个聊天渠道**。因为“能启动”只说明后端活着；“能收到消息并回消息”，才说明你开始拥有一条真正可用的工作链路。
