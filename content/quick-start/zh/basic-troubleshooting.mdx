第一次运行失败时，最快的路径不是乱改配置，而是先把症状缩小。对新手来说，最危险的不是某个真正难解的 bug，而是把原本简单的问题改复杂了。

## 第一轮检查

先跑最短的验证环：

- 进程是不是在运行？
- 模型凭证是不是有效？
- 渠道是不是正确配对？
- 日志里有没有明确报错？

这四个问题几乎覆盖了 Quick Start 阶段的大多数失败原因。先把它们查完，再谈更深层的推理。

## 怎么避免随机排错

建议固定用这个顺序：

1. 先确认你看到的现象
2. 看离现象最近的日志或健康信号
3. 一次只改一个变量
4. 改完立刻重试一次

这通常就能看出问题更接近 setup、渠道、凭证，还是 workflow 范围本身。

排错时最有价值的不是“多懂一点”，而是**顺序稳定**。顺序一旦稳定，你就能越来越快地判断问题属于哪一层。

## 先给问题分层

大多数 first-run 问题，其实都能归到下面几层：

- 安装层：命令跑不起来、服务没启动、环境不完整
- 模型层：key 无效、provider 配错、实际调用模型和预期不一致
- 渠道层：机器人没收到消息、权限不足、账号未绑定
- 工作流层：触发进来了，但流程设计太大或依赖太多

你不需要一开始就知道怎么修所有问题，但至少要先知道问题大概落在哪一层。

## 最短排错环路是什么

当你不知道该从哪下手时，回到这套最短环路：

1. 看服务是否正常存活
2. 看最近日志是否有硬错误
3. 做一个最小动作复现问题
4. 只改一个变量再重试
5. 判断问题有没有移动

所谓“问题有没有移动”很重要。比如你原来是“消息没回来”，改完后变成“消息回来了但内容报错”，这其实是进展，不是失败。说明你已经穿过了上一层问题。

## 新手最常见的误区

一失败就同时改 prompt、模型、环境和技能，最后把唯一有价值的线索也一起改没了。

另一个常见误区是过早下结论：

- 机器人沉默，就认定是模型坏了
- 模型报错，就认定是平台 bug
- 一个技能失败，就认定整个系统不稳定

Quick Start 阶段，很多问题都只是链路没补齐，不需要上升到“系统有问题”。

## 什么时候该停下继续猜，转去看专门指南

如果你已经能明确说出下面这种句子，就不要继续泛化排错了：

- “进程是活的，但飞书消息收不到”
- “渠道通了，但 provider 认证失败”
- “基础对话能回，但 starter workflow 一触发就报错”
- “同样的命令昨天能跑，今天只有某个技能失败”

一旦症状足够具体，就应该进入对应的 Troubleshoot 路径，而不是继续停留在 Quick Start 的泛排错阶段。

## 一个最重要的习惯

每次排错都尽量留下这三个信息：

- 你做了什么动作
- 你看到什么现象
- 你刚刚改了什么变量

这会极大降低“我好像试过很多，但忘了哪些真的试过”的混乱感。很多排错失败，并不是因为问题太难，而是因为记录太差。

## 更好的下一步

当你已经能明确描述症状时，就应该跳到对应的 Troubleshoot 路径，沿着匹配的恢复指南继续推进。Quick Start 的作用，是帮你从“完全没头绪”走到“我知道问题在哪一层”。
