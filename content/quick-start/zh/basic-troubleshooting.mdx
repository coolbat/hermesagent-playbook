Hermes 早期失败大多不神秘，通常都落在四层之一：

- 安装层
- provider 配置层
- gateway 或平台路由层
- 工作流范围层

## 最短恢复链路

出问题时，先走这条最短但有用的链路：

```bash
hermes setup
hermes model
hermes doctor
```

如果 CLI 正常，但平台里不正常，再加一条：

```bash
hermes gateway
```

这样最容易把“运行时本身坏了”和“平台接线坏了”分开。

## 先按症状排，不要先按猜测排

从症状出发，而不是从你最喜欢的理论出发。

常见情况：

- CLI 没响应：先查安装和 provider
- CLI 正常、平台沉默：先查 gateway 和平台路由
- 回复质量很差：先查模型选择和工作流范围
- 执行动作不对：先查 skills、规则和边界

## 最容易浪费时间的做法

尽量不要在早期排错时做这些：

- 同时切换 provider 和渠道
- 在排错过程中继续装新技能
- 基础链路没验证就先改 prompt
- 把一次失败升级成一次全面重装

变量越多，信号越差。

## 先用 Hermes 自带诊断，再讲理论

如果 Hermes 已经给你诊断命令，先用它，再猜。

官方的排查顺序本质上很务实：

- 先确认环境状态
- 再确认 provider 状态
- 再确认路由状态
- 最后才考虑重做工作流

这个顺序会节省很多时间。

## 什么时候进入更深层排查

当你已经能准确描述问题时，就可以离开 Quick Start：

- “provider 鉴权失败”
- “gateway 正常但 Telegram 沉默”
- “工作流有返回，但价值很低”

到了这个粒度，Troubleshoot 和 Learn 才能真正帮上忙。
