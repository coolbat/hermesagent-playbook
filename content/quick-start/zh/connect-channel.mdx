CLI 证明 Hermes 能工作。接通渠道，才证明 Hermes 能以真实用户会使用的方式被调用。

## 先接一个平台就够了

第一步最适合选一个测试成本低的平台：

- Telegram
- Discord
- 其他 Hermes 官方支持的消息表面

不要一上来接多个。只要失败了，你就希望事件路径足够单一，方便定位。

## 顺序一定要简单

官方文档里反复强调一个顺序：

1. 先确认 Hermes 文本链路是好的
2. 再接一个平台
3. 验证一条请求-响应闭环
4. 最后再加 voice、automation 或更多渠道

这个顺序的价值，是把故障范围控制在最小。

## 一次好的渠道测试长什么样

第一次测试应该非常朴素：

- 一个明确的账号
- 一条干净的文本消息
- 一次明确的返回

你此时还不是在测试“智能够不够强”，而是在测试消息有没有进入 Hermes、Hermes 有没有处理、结果有没有回到正确位置。

## 如果你走的是 gateway 路径

平台配置完成后，启动 gateway：

```bash
hermes gateway
```

对于 gateway 驱动的平台来说，这个进程和 provider 一样关键。CLI 健康但 gateway 不健康，用户侧仍然是坏体验。

## 如果机器人沉默

按这个顺序查：

1. 平台配置是否正确
2. gateway 是否在运行
3. Hermes 在 CLI 里是否仍然正常
4. 日志里有没有看到平台事件

如果第 3 步不正常，先回到安装与 provider。第 3 步正常、第 4 步没有事件，大多是 webhook、鉴权或平台路由的问题。

## 第一次别混进太多变量

初次验证时，不要混这些东西：

- 附件
- 富文本
- 语音消息
- 很重的工具调用 prompt

纯文本最容易看清路径，也最容易恢复。
