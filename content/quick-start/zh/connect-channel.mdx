安装只证明系统能启动。接通渠道，才证明系统能**接收、路由并回答**。这一步非常关键，因为它第一次把“后台里的 OpenClaw”变成“你真的能用的 OpenClaw”。

## 什么算成功

你通过一个真实渠道发出测试消息，并得到一次真实响应。

这件事虽然简单，但能同时确认很多关键点：

- 机器人可以被触达
- 渠道绑定是通的
- 消息路径是成立的
- 运行时行为可以被观察

如果你发出去一条消息、收到回来一条消息，你就已经完成了 Quick Start 里最重要的里程碑之一。

## 第一个渠道尽量简单

先选最容易测试和重试的渠道，例如：

- Telegram
- 飞书
- 其他已经支持、且你熟悉的聊天入口

判断标准不是“哪个最强”，而是“哪个最容易观察到输入、输出和错误”。第一次接渠道，不要优先选最复杂、最需要企业权限、最难登录排障的方案。

## 推荐的连接顺序

把这一步拆得足够简单：

1. 选一个你最熟悉的聊天渠道。
2. 只配置一个账号，不要同时配多个。
3. 完成绑定后，立刻发送一条最短测试消息。
4. 看响应有没有回来，再看日志有没有记录。
5. 成功后再考虑频道权限、群聊接入或多账号场景。

第一次测试消息也尽量简单。不要一上来就让它联网搜索、读文件、调技能。先测试“普通消息能回来”这一件事。

## 一条测试消息，到底在验证什么

这一步不只是验证“机器人会回话”，它其实在验证一整段链路：

- 渠道侧配置正确
- OpenClaw 可以接到消息事件
- 关联账号或 bot token 有效
- 消息被正确路由到对应会话
- 模型调用没有在最短路径上失败
- 结果能够再送回渠道

所以这一步越早做越好。因为它能尽快排除掉大量不确定性。

## 如果机器人没反应，先检查这几类问题

很多“渠道问题”其实并不复杂，通常集中在下面几类：

- 账号或 token 配错
- 回调、事件订阅或权限没开
- 机器人加进了渠道，但没有拿到收发权限
- OpenClaw 进程没在正常运行
- 模型调用失败，导致看起来像“消息没进来”

先把问题缩成“消息没进来”还是“消息进来了但没回去”，排错速度会快很多。

## 新手最常见的误区

当机器人没反应时，很多人第一反应是继续加技能、改 prompt，甚至换模型。通常这只会把问题越改越糊。渠道闭环没通之前，功能扩展几乎没有意义。

另一个常见误区，是把“能在某个后台页面看到 connected”误当作真正成功。真正成功的标准只有一个：**你在真实聊天界面发出消息，机器人真的回了。**

## 这一页完成后，你应该得到什么

继续往下之前，最好确认下面几件事：

- 你已经选定了一个主渠道作为第一入口
- 你已经完成了一次真实消息往返
- 你知道渠道失败时该先查账号、权限还是日志
- 你还没有被多渠道、多群、多账号复杂度拖住

做到这里，OpenClaw 才算从“装在电脑里”变成“开始能被调用”。

## 更好的下一步

如果渠道已经通了，就进入**第一个入门工作流**。如果渠道没通，先把它当成渠道问题，而不是 workflow 问题。不要还没收发成功，就提前优化工作流设计。
