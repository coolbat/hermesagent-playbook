Gateway 是把本地健康的 Hermes 运行时变成“外部可调用闭环”的那一层。如果你只在 CLI 里用 Hermes，它还不是重点；但一旦你要接 Telegram、Discord 或其他平台，gateway 就会进入关键路径。

## Gateway 到底做什么

可以把 gateway 理解成 Hermes 和外部平台之间的路由层：

- 接收平台事件
- 转发给 Hermes 运行时
- 把输出送回正确目标

所以 gateway 出问题时，看起来会像是 Hermes 坏了，但实际核心运行时也许完全正常。

## 什么时候需要它

你通常在这种情况下需要关注 `hermes gateway`：

- CLI 正常
- provider 正常
- 但外部平台消息始终走不完整个闭环

如果你的所有流程都只在终端里，gateway 还不是第一优先级。

## 正确排查顺序

不要一上来就盯平台。先从核心运行时开始：

1. 确认 Hermes 在 CLI 里正常
2. 确认 provider 仍然能返回
3. 启动或重启 gateway
4. 发一条干净的平台消息
5. 把平台表现和 gateway 日志对照起来看

这个顺序能很快告诉你问题在运行时还是在平台路由。

## 第一次测试要尽量干净

第一次 gateway 测试只保留：

- 一个平台
- 一个账号
- 一条简单文本
- 一次明确回复

这样即使失败，信号仍然足够干净。

## 最常见的误判

最容易出错的地方是把“进程还活着”当成“gateway 已经健康”。

一个活着的 gateway 仍然可能：

- 读取了错误环境
- 指向过期回调
- 路由到错误目标
- 在平台边界静默失败

所以 gateway 排查本质上是“路由排查”，不只是“进程排查”。
