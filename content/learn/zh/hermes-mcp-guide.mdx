MCP 之所以强，是因为它能明显扩大 Hermes 能接触到的外部能力。这也是为什么它不该成为你加的第一层复杂度。

## MCP 真正改变了什么

普通 skills 往往封装的是一条相对明确的能力路径。MCP 则是通过协议边界，把 Hermes 接到外部工具或服务上。

这意味着 MCP 能带来真实杠杆，但也会带来：

- 更多环节
- 更多权限边界
- 更多失败面

## 什么情况下值得上 MCP

当这几件事同时成立时，MCP 才更值得：

- 基础 Hermes 流程已经稳定
- 你要的能力确实来自外部系统
- 这个能力带来的结果值得再引入一层集成复杂度

如果你还在验证基础运行时，MCP 通常不是好起点。

## 更稳的接入顺序

更安全的顺序通常是：

1. 先证明 CLI 闭环
2. 先证明没有 MCP 时工作流也成立
3. 只加入一个 MCP 能力
4. 单独测试它
5. 再把它并回更大的工作流

这样失败更容易归因。

## 最关键的区别

如果一条工作流本来就不稳定，MCP 不会让它更清楚，只会让诊断更难。

正确用法是：用 MCP 扩展一条健康工作流，而不是拿它去拯救一条不健康工作流。

## 出问题时先看什么

当 MCP 相关行为失败时，先确认：

- Hermes 自身仍然健康
- 外部能力本身可达
- 任务真的在调用预期能力

不要一上来就重做整条工作流。
