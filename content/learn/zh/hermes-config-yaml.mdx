配置文件不只是“设置堆放处”。在 Hermes 里，它本质上是控制 providers、agents、tools 和运行时行为如何连接起来的控制面。

## 这个文件应该帮你回答什么

当 `config.yaml` 是健康的，你应该能回答：

- Hermes 现在在用哪个 provider
- 当前启用了哪些 agent 默认行为
- 哪些 tools 或 integrations 已开启
- 哪些运行时行为是有意设计的，哪些只是意外堆出来的

如果这个文件看起来像魔法，那说明它还太不透明。

## 最好按层来读

更实用的阅读方式是自上而下：

1. provider 和 model 默认值
2. agent 级行为
3. tool 与 integration 配置
4. memory、delegation 这类可选运行时特性

不要一次改五层。配置漂移最难排的，就是多个关注点同时变化。

## 把它当成单一事实来源

最常见的问题是：真实配置一半藏在 shell 里，一半散落在笔记里。好的配置文件应该让运行时形态本身就是可读的。

这不意味着所有东西都该写进文件。密钥仍然要按正确方式管理。但系统结构应该一眼能看明白。

## 什么叫好的配置

好的配置应该是：

- 范围窄
- 能解释
- 和你当前真正运行的工作流一致

坏的配置通常是：

- 从别的环境复制来的
- 塞满当前根本没用的选项
- 一出问题就继续乱改

## 什么时候才该改配置

只有当你能明确说出原因时，再改 `config.yaml`：

- “我要换一个 provider 默认值”
- “我要启用一个运行时能力”
- “我要调整一个 agent 行为”

如果原因只是“感觉哪里不对”，先诊断，不要先改。
