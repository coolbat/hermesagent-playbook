## 这通常意味着什么

基础运行时大概率是健康的，问题更可能在面向平台的路由层。

## 第一轮恢复动作

先做最窄的对照测试：

1. 确认 Hermes 在 CLI 中仍然正常
2. 重启 `hermes gateway`
3. 发一条干净的平台消息
4. 对照发送时间和 gateway 日志窗口

如果 CLI 正常、平台不正常，就把注意力放在路由和平台状态上。

## 接下来该核对什么

按这个顺序查：

- gateway 是否读取了预期环境
- 平台回调或 webhook 是否仍然有效
- 目标账号或路由是否正确

在这些没弄清之前，不要先重做工作流。

## 如果问题还在

把测试面缩到最小：一个平台、一个账号、一条消息、一个预期回复。测试越大，噪音越多。
